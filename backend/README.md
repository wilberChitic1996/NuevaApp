# Cadejo — Daily Leaderboard Backend (Supabase)

> **These are scripts to review and run yourself.** Nothing here is executed by
> the app or by any agent. No credentials are committed.

## What validates where

| Concern | Client (app) | Server (Edge Function) |
|--------|--------------|------------------------|
| Which puzzle | Derives daily seed from UTC date (`DailySeed`) | **Re-derives the seed from the date** — ignores any client seed |
| Score | Computed locally for display only | **Recomputed by replaying the intent tape** — the only score trusted |
| Genuine app/device | Requests a Play Integrity token | **Verifies the token** before accepting |
| Writing the row | Cannot (RLS denies) | Writes with the service-role key after validation |

The client sends `{ date, intents, score, reached_level, integrity_token }`. The
server trusts only `date`, `intents` and `integrity_token`; it recomputes `score`
and `reached_level` and rejects the submission if they don't match.

## Files

- `supabase/migrations/0001_daily_leaderboard.sql` — table, indexes, RLS.
- `supabase/functions/submit-daily-score/index.ts` — the validating endpoint.
- `supabase/functions/submit-daily-score/engine.ts` — TypeScript port of the
  deterministic engine (mirrors `:core:domain`).

## Deploy (you run this)

```bash
# 1. Apply the schema
supabase db push          # or paste the SQL into the Supabase SQL editor

# 2. Deploy the function
supabase functions deploy submit-daily-score

# 3. Set secrets the function needs (service role bypasses RLS to insert)
supabase secrets set SUPABASE_URL=... SUPABASE_SERVICE_ROLE_KEY=...
# For Play Integrity, add your Google service-account config and implement the
# TODO in verifyIntegrity(); during local testing you may set:
supabase secrets set ALLOW_MISSING_INTEGRITY=true   # DEV ONLY — never in prod
```

Then put the project URL + anon key into the app's `local.properties`
(`SUPABASE_URL`, `SUPABASE_ANON_KEY`) or CI secrets — see `local.properties.template`.

## Play Integrity — what to finish before production

`verifyIntegrity()` in `index.ts` is a deliberate extension point. In production it
must call the Play Integrity API with your service account and require:
- `appIntegrity.appRecognitionVerdict == PLAY_RECOGNIZED`
- `deviceIntegrity` contains `MEETS_DEVICE_INTEGRITY`
- (optional) `accountDetails.appLicensingVerdict == LICENSED`

Until that's implemented the function rejects tokenless submissions (safe default).

## ⚠ Cross-validate the engine port before going live

The TS port must produce **identical** numbers to the Kotlin engine, or it will
reject legitimate scores. A golden vector is generated by the Kotlin test
`GoldenVectorTest` (run `./gradlew :core:domain:test --tests "*GoldenVectorTest*"`,
it prints seed/score/status/level for date `2026-07-04` and a fixed tape). The
matching Deno test lives at `supabase/functions/submit-daily-score/engine_test.ts`
— fill its expected values from the Kotlin output and run `deno test` to confirm
they agree. Do this whenever either engine changes.

## Data Safety (Play Console)

What this backend collects: a **display name** (user-provided), a **score**, the
UTC **date**, and a **non-identifying device hash** (SHA-256 truncation, not an
advertising ID or hardware ID). No location, contacts, or personal identifiers.
Reflect exactly this in the Play Console Data Safety form.
